## Context

The vault embedder is FNV-1a bag-of-words (`bin/smdjad/src/embedder.rs`): `embed(text)` lowercases whitespace tokens, hashes each into one of `DIM = 128` buckets, accumulates counts, and L2-normalises so cosine equals the dot product. It is fast, deterministic, dependency-free, and offline — but purely lexical. Two texts score highly only when they share literal words; synonyms and paraphrases score near zero. Recall is correspondingly weak on every path the embedder backs: cold-context retrieval (`bin/smdjad/src/orchestrator/cold.rs`), the `smedja_vault_search` tool (`bin/smdjad/src/executor/mod.rs:451`), lean-specs detail recall (`bin/smdjad/src/lean_spec.rs`), and the `compact` summary namespace (`bin/smdjad/src/main.rs:2037`).

The vault stores each embedding as a little-endian `f32` BLOB (`bytemuck::cast_slice`) and ranks by cosine over a full table scan (`crates/smedja-vault/src/vault.rs` — `search` at :289, `query` at :504; `similarity::cosine_sim` at `crates/smedja-vault/src/similarity.rs:12`). Cosine over two vectors is only meaningful when both came from the **same model** and have the **same dimension**. The schema already carries the seed of model-awareness: `EmbedderIdentity { model, dimensions }` is persisted in `vault_meta` and `Vault::insert` rejects a dimension-mismatched embedding with `VaultError::EmbedderMismatch` (`vault.rs:63,158`). But that guard is vault-global (one identity per database) and is never set by the daemon; `search`/`query` still compare every stored BLOB against the query vector. The orchestrator already proves the swappable-backend pattern with the `ColdStore` port (`crates/smedja-memory/src/cold.rs`): a trait the memory crate owns, with the daemon supplying a concrete `VaultColdStore` adapter — the embedder should follow the same shape.

## Goals / Non-Goals

Goals:
- Define an `Embedder` port (trait: `embed`, `model_id`, `dim`) and route every vault-embedding call site through it.
- Add a learned `/v1/embeddings` backend that sharpens semantic recall when reachable.
- Keep FNV-1a as the named default and the always-available offline fallback; never hard-fail on a missing learned model.
- Tag every vault row with its producing `model_id` + `dim`; make `search`/`query` compare only same-model vectors.
- Provide a re-embed/backfill path to upgrade existing FNV rows into the learned model space.

Non-Goals:
- Replacing the hybrid keyword + recency scoring in `Vault::search` — only the cosine term changes meaning; the keyword/recency boosts stay.
- An approximate-nearest-neighbour index — the full-scan cosine is retained (cold stores are small per the `query` doc comment, "< ~10,000 entries").
- In-process model weights / GPU placement — smedja orchestrates external serving (see Decision 2); it does not load weights or download models itself.
- Fine-tuning or training an embedding model.
- Cross-model comparison or vector projection between dimensions — incomparable rows are excluded, not reconciled mathematically.

## Decisions

**Decision 1: An `Embedder` port, mirroring `ColdStore`.**
Introduce a trait — `Embedder { fn embed(&self, text: &str) -> Vec<f32>; fn model_id(&self) -> &str; fn dim(&self) -> usize; }` — owned where the daemon can supply concretions (the daemon, alongside `embedder.rs`), exactly as the daemon supplies `VaultColdStore` for the memory crate's `ColdStore` port. The FNV-1a function stays but is wrapped by an `FnvEmbedder` implementation reporting `model_id = "fnv-bow-128"`, `dim = 128`. The daemon resolves one `Arc<dyn Embedder>` at startup and threads it to every call site (`cold.rs`, `executor/mod.rs`, `lean_spec.rs`, `main.rs`, `handlers/*`). The learned `/v1/embeddings` call is network I/O, so the trait method that the daemon actually invokes on the live path is async; the cosine math and tests stay synchronous.
- Rationale: one swap point, one source of truth for `model_id`/`dim`, and it matches the established port pattern so reviewers recognise it immediately.
- Alternative — a free function with a runtime `match`: rejected; it scatters the backend choice across nine call sites and gives no place to carry `model_id`/`dim`.

**Decision 2 (model placement): RECOMMEND a local OpenAI-compatible `/v1/embeddings` endpoint.**
Three options were weighed:
- **(a) Local `/v1/embeddings` endpoint (RECOMMENDED).** Reuse the existing local-runner path: `crates/smedja-adapter/src/local.rs` already health-checks `GET /v1/models` against `SMEDJA_LOCAL_ENDPOINT` and `OpenAiProvider` already speaks `/v1/chat/completions` (`crates/smedja-adapter/src/openai.rs:98`). An embeddings client is the same shape — `POST {endpoint}/v1/embeddings` with `{ "model", "input" }`, parse `data[].embedding`. This keeps the **orchestrate-not-serve** boundary the local-model-ux work established ("smedja orchestrates the external local-serving tools … it does not reimplement an inference server", `local.rs:5`): no heavy in-process dependency, no weights in the daemon, offline-capable when the local stack is up, and the existing `reqwest`/timeout/health-check machinery is reused.
- **(b) In-process Rust model (`fastembed`/`ort` ONNX, or `candle` + MiniLM).** Pure-Rust and self-contained once the model is present, but pulls a heavyweight dependency (ONNX runtime or candle + tokenizers) and a model-download/cache step into the daemon, contradicting orchestrate-not-serve and bloating the binary. Deferred — could be revisited as an additional `Embedder` impl if a no-external-process deployment is ever required.
- **(c) Provider embeddings API (e.g. OpenAI `text-embedding-3-*`).** Best raw quality, zero local infra, but the vault embeds a **lot** — every cold/compact summary, every lean-spec chunk, every `smedja_vault_search` query, plus a full re-embed of the existing corpus on backfill — so per-embed network cost and latency add up fast, and it breaks offline operation. Flagged on cost; kept available as a config-selectable backend but not the default.

Recommendation: ship the local `/v1/embeddings` backend behind the `Embedder` port, with FNV-1a as the fallback. This is consistent with how local chat models are already wired, adds no heavy in-proc dependency, and keeps the daemon offline-capable. The endpoint and model id come from `[embedder]` in `.smedja/config.toml`; an unreachable endpoint at startup (the existing 500 ms health check) resolves the port to `FnvEmbedder` instead of failing.

**Decision 3 (THE CRUX): per-row model/dimension versioning + re-embed/backfill.**
FNV is `dim = 128`; a learned MiniLM is 384, others 768 / 1536. Vectors from different models or dimensions are **not comparable** — cosine between a 128-d FNV vector and a 384-d MiniLM vector is meaningless even where the lengths happen to align, and `cosine_sim` (`similarity.rs:12`) silently zips to the shorter slice rather than erroring. The vault's current global `EmbedderIdentity` guard only blocks *inserts* of the wrong dimension; it does nothing for a corpus that already mixes models, and it forces a destructive choice (one model per whole database). The fix is per-row tagging plus same-model-only comparison:

1. **Tag each row.** Add `embedder_model_id: String` and `dim: usize` to `VaultEntry` (`vault.rs:8`), persist them as new `vault_entries` columns (with idempotent `ALTER TABLE ... ADD COLUMN` migrations matching the existing pattern at `vault.rs:637`), and write the active embedder's `model_id`/`dim` on every `insert`/`upsert`.
2. **Compare only same-model vectors.** `Vault::search` and `Vault::query` take (or derive) the query's `model_id`/`dim` and, in the scan, **skip** any row whose `embedder_model_id` ≠ query model or `dim` ≠ query dim. Skipped rows are excluded from ranking — never fed to `cosine_sim`, never an error. This replaces the brittle `DimensionMismatch` error in `query` (`vault.rs:527`) for the mixed-corpus case: a mismatched row is simply not a candidate, so a vault mid-migration keeps returning correct same-model results instead of erroring out.
3. **Re-embed / backfill.** Add a `vault.reembed` RPC (and an `smj` command) that iterates rows in a namespace (or whole vault), re-embeds each row's stored `content` with the active learned `Embedder`, and rewrites the embedding + `embedder_model_id` + `dim`. This is the explicit upgrade path for the existing FNV `compact`/lean-spec corpus; it is restartable and idempotent (re-embedding a row already at the active model is a no-op write). Lazy re-embed-on-access was considered but rejected as the primary mechanism: it leaves the corpus permanently mixed (cold queries against not-yet-touched rows keep missing) and complicates the read path; an explicit backfill converges the whole namespace to one comparable space. The global `EmbedderIdentity` is retained as a coarse "what does this vault mostly hold" marker and is updated to the active model after a successful full backfill.

- Rationale: cosine is only valid within a model space; the only correct cross-model behaviour is exclusion plus an explicit migration, not silent comparison or a hard error.
- Trade-off: during a partial backfill the vault holds two model spaces and a query only sees its own; recall on un-migrated rows is temporarily reduced (to zero for that query) rather than wrong. Acceptable and self-healing once backfill completes.

**Decision 4: backward / offline behaviour — never hard-fail.**
With no learned endpoint configured or reachable, the daemon resolves the `Embedder` port to `FnvEmbedder` and everything works exactly as today (weaker recall, same correctness). A learned endpoint that becomes unreachable mid-session must not crash a turn: the live embed call surfaces the failure as "fall back to FNV for this call" or "return empty results", never a panic or a turn-aborting error — matching the local-runner health-check degradation (`local.rs` health check sets `healthy = false` and the tier is skipped). Stored learned-model rows simply become un-queryable from an FNV query (Decision 3 excludes them), which is the correct same-model-only behaviour, not a failure.

## Risks / Trade-offs

- [Risk] Mixed-model corpus during/after partial backfill reduces recall for the active query model → Mitigation: same-model-only comparison keeps results *correct* (never wrong), and the `vault.reembed` backfill converges the namespace; document running it after a backend switch.
- [Risk] Learned endpoint latency on the hot path (cold retrieval / `smedja_vault_search` run per turn) → Mitigation: the embed call is async and off the cosine path; an unreachable/slow endpoint degrades to FNV via the existing health-check timeout rather than blocking.
- [Risk] Schema change to `vault_entries` could break old databases → Mitigation: additive columns via idempotent `ALTER TABLE ... ADD COLUMN` (the established `vault.rs:637` pattern); pre-existing rows default `embedder_model_id` to the FNV id and `dim` to the stored BLOB length / 4, so legacy rows remain correctly same-model-comparable.
- [Risk] Provider-API backend (option c) cost blowout on backfill → Mitigation: it is not the default; cost is flagged; backfill batches and is restartable so it can be paused.
- [Risk] `cosine_sim` silently zipping mismatched lengths masks a model mix → Mitigation: callers in `search`/`query` filter by `model_id`/`dim` *before* calling `cosine_sim`, so the function is only ever handed same-dim vectors on the live path.
